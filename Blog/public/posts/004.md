## Les branches Git

### Concept des branches

Les branches sont utilisé afin de coder en parallèle avec des collaborateurs. Elles sont visible comme des "lignes de développement". Elles permettent de faire des commits en ne changeant que sa "ligne" sans impacter le travail des autres. 

### Les branches dans un projet

A la création d'un dépot git, une branche est créer, la branche `main` ou `master`. C'est cette branche qui contient le code qui est executé. Cela veut dire que ce qui est ajouter sur la branche `main` ne doit en aucun cas générer d'erreurs ou entrer en conflict avec du code existant car, soit le logiciel est en cours de conception et tous les autres collaborateurs ne pourront plus travailler à cause de cette erreur, soit le logiciel est déja déployé et cela causerait le crash du logiciel. Pour éviter cela, chaque collaborteur créer sa branche. Comme cela, s'il génère une erreur il est le seul impacté. Après avoir vérifié que son code est compatible avec la branche main, le développeur peut ajouter les modifications effectué sur sa branche à la branche `main`, c'est ce qu'on appelle un `merge`.  

Afin de nommer les branches, aucune règles n'est imposé par **Git**, mais des nomenclatures sont utilisé en entreprise afin de comprendre quel est l'intéret d'un branche. Cette nomenclature est `type-de-changement/fonction-en-elle-même`.  
Les types de changement souvent utilisé sont les suivant :

| Branche | Rôle  |
| ----------- | ------------------------------------------------------ |
| `main` | Représente la version stable en production. Tout ce qui y est fusionné doit être prêt à être déployé. |
| `develop` | Intégration des nouvelles fonctionnalités. Cette branche sert de base pour toutes les releases futures. |
| `feature/*` | Chaque nouvelle fonctionnalité se développe sur une branche `feature` issue de `develop`. |
| `release/*` | Branches temporaires pour préparer une version à publier. Elles permettent de stabiliser le code, corriger des bugs et mettre à jour la documentation. Elles partent de `develop` et sont fusionnées à la fois dans `main` et `develop`. |
| `hotfix/*` | Branches pour corriger rapidement un bug en production. Elles partent de `release` et sont fusionnées à la fois dans `main` et `develop`. |

Tant qu'a la parti de la fonction en elle même elle décrit ce que fit concrètement la fonction comme par exemple `login`, `add-payment-method`, etc... Il est préférable d'utiliser des tirets plutôt que des espaces ou des underscores et d'éviter les caractères spéciaux et les accents.

### Creér une branche

Pour créer une branche, rentrer la commande suivante dans le terminal :  
``` bash
git switch -c feature/fonction-nouvelle-feature
```

L'option `-c` permet de créer une nouvelle branche. Sans option, switch permet juste de changer de branche.

### `git switch` vs `git checkout`

Avant l’introduction de git switch, la commande utilisée pour changer de branche était :

``` bash
git checkout nom-de-branche
```

Et pour créer et changer de branche :
``` bash
git checkout -b nom-de-branche
```

#### Pourquoi git switch a été créé ?
La commande git checkout faisait trop de choses différentes :

- Changer de branche `git checkout nom-de-branche`
- Restaurer un fichier `git checkout main.py`
- Revenir à un commit `git checkout a1b2c3d`
- Créer une branche `git -b checkout nom-de-branche`

Cela la rendait parfois confuse, surtout pour les débutants.

Depuis Git 2.23 (2019), les développeurs de Git ont introduit :

- `git switch` -> pour gérer les branches
- `git restore` -> pour restaurer des fichiers

## La réalité du développement en entreprise

Dans une entreprise, les développeurs ne travaillent généralement pas "librement" sur ce qu’ils veulent. La majorité des équipes travaillent en méthodologie agile, souvent en Scrum. Scrum est un cadre de travail agile structuré autour des sprints, d'un backlog, et des ***tickets***.

### Les tickets
Un ticket est une unité de travail qui peut représenté plusieurs chose : Une nouvelle fonctionnalité, Un bug à corriger, Une tâche technique.
Des outils comme [Jira](https://www.atlassian.com/software/jira) sont très utilisés en entreprise pour gérer ces tickets.

![jira board](/images/jira_board.png)

### Les différents niveaux de tickets
Dans des outils comme Jira tous les tickets n’ont pas la même taille ni le même rôle. On distingue généralement plusieurs niveaux :

#### Les Stories
Une Story est une fonctionnalité plus petite et concrète.

- Réalisable par un seul développeur
- Généralement terminée en un sprint
- A un ***epic*** comme parent

Exemple de ticket :
- ISICG-142 – Ajouter le shadow map

#### Les Epics
Un Epic représente une grande fonctionnalité ou un objectif majeur du projet.

- Trop volumineux pour être réalisé par une seule personne
- Peut durer plusieurs sprints
- Regroupe plusieurs stories

**Exemple :**
ISICG-128 – development du level editeur

Il pourra contenir les tickets stories suivant :
``` text
Epic: ISICG-128 development du level editeur
 ├── ISICG-129 – Créer un système de grille
 ├── ISICG-130 – Ajouter/supprimer des objets de scène
 ├── ISICG-131 – Outil de transformation d'objets
 ├── ISICG-132 – Enregistrer le niveau au format JSON
 ├── ISICG-133 – Charger le niveau existant
 └── ISICG-134 – Interface utilisateur de l'éditeur de base
```

Un epic est donc une vision globale, découpée en tâches plus petites.

#### Les Bugs

Un Bug correspond à un comportement incorrect du logiciel.
- Corrige un problème existant
- Peut être critique ou mineur
- Peut être rattaché à une story ou un epic

Exemple :
ISICG-87 – mal fonctionnement de la FOV

### Workflow réel – Exemple ISICG-128

Nous allons prendre l’Epic :
- ISICG-128 – Development du level editor

Et voir comment une équipe pourrait réellement organiser les branches pendant un sprint.

#### Début du sprint – Création de la branche Epic
Au début du sprint, le chef d'équipe crée une branche dédiée à l’Epic.
Cette branche part généralement de develop (***et non de main!***).

``` bash
git switch develop
git fetch 
git rebase origin/develop
git switch -c feature/ISICG-128-level-editor
```

L’Epic aura donc sa propre branche de regroupement.

#### Les développeurs créent leurs branches depuis l’Epic
Chaque développeur prend une Story liée à l’Epic.

Exemple :
- ISICG-129 – Grid system
- ISICG-130 – Add/remove objects
- ISICG-132 – Save level JSON

Ils créent leur branche à partir de la branche Epic :

``` bash
git switch feature/ISICG-128-level-editor
git fetch 
git rebase feature/ISICG-128-level-editor
git switch -c feature/ISICG-129-grid-system
```

Un autre développeur :

``` bash
git switch feature/ISICG-128-level-editor
git fetch 
git rebase feature/ISICG-128-level-editor
git switch -c feature/ISICG-132-save-json
```

Un autre développeur :

``` bash
git switch feature/ISICG-128-level-editor
git fetch 
git rebase feature/ISICG-128-level-editor
git switch -c feature/ISICG-130-add-remove-objects
```

Les branches Story partent de la branche Epic, pas directement de develop.

Cela permet :
- D’isoler la fonctionnalité complète
- De tester l’Epic indépendamment
- D’éviter d’impacter les autres features en cours

#### Merge des Stories dans la branche Epic
Quand une Story est terminée :
- Pull Request vers feature/ISICG-128-level-editor
- Code review
- Merge

#### Merge final vers develop
Une fois :
- Toutes les Stories mergées
- Les tests validés
- Les bugs corrigés

Le team lead merge l’Epic vers develop.

#### Vue globale du workflow

``` text
develop
   │
   ├── feature/ISICG-128-level-editor
   │        ├── feature/ISICG-129-grid-system
   │        ├── feature/ISICG-130-add-objects
   │        └── feature/ISICG-132-save-json
   │
    ...
```
